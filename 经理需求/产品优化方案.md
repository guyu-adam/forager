# Forager 产品优化方案

> 产品经理视角 · 2026-05-26
> 目标：实现"爬取需求 → 解决需求 → 上传交付 → 收款"的商业闭环

---

## 一、当前项目评估

### 1.1 现有模块

| 模块 | 完成度 | 评价 |
|---|---|---|
| `demand_radar.py` | 60% | 爬取+评分已完成，但仅覆盖4个平台，评分规则粗糙，无需求分类 |
| `gpu_earner.py` | 20% | 仅有状态查看和手动扫描，商业闭环完全缺失 |
| `run.sh` | 10% | 只有启动脚本，无部署/监控/告警 |

### 1.2 缺失的核心能力

```
爬取需求 ✅ → 解决需求 ❌ → 上传交付 ❌ → 收款 ❌
```

现有项目只是一个"需求雷达"，不是一个"觅食者"。Forager 的名字意味着自动出去找食物带回来——但现在只有"找"，没有"带回来"。

## 二、商业闭环设计

### 2.1 核心商业模式

**定位**：AI 驱动的开源赏金自动化解决器

**一句话**：自动发现 GitHub 上的付费 issue，用 AI 解决，自动提 PR，收到赏金。

**为什么选 GitHub 为主战场**：

| 维度 | GitHub | 传统 freelancer 平台（猪八戒/Upwork） |
|---|---|---|
| 交付方式 | 自动提 PR，无需人工交互 | 需要沟通、报价、谈判 |
| 支付确定性 | 平台托管（Algora/Opire），merge 即付 | 需客户确认，纠纷风险高 |
| API 可编程性 | 完整 REST API + GraphQL | 无公开 API，反爬严格 |
| 竞争形态 | PR 质量竞争，可以差异化 | 价格战，只能拼低价 |
| 自动化可行性 | 高 | 低 |

### 2.2 商业闭环三步走

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  1. 发现      │ ──→ │  2. 解决      │ ──→ │  3. 交付      │
│  Find        │     │  Solve       │     │  Deliver     │
├──────────────┤     ├──────────────┤     ├──────────────┤
│ · GitHub API │     │ · AI 理解需求 │     │ · 自动 fork   │
│ · 赏金平台聚合│     │ · 生成代码    │     │ · 自动写代码  │
│ · 智能评分    │     │ · 本地测试    │     │ · 自动提 PR   │
│ · 竞争分析    │     │ · 验证通过    │     │ · 自动回复    │
└──────────────┘     └──────────────┘     └──────────────┘
       ↑                                        │
       └──────────── 4. 收钱 ←─────────────────┘
                  · Algora/Opire 自动结算
                  · 多平台赏金聚合
```

## 三、需求发现层优化（Find）

### 3.1 赏金平台聚合（新增）

当前仅搜 GitHub issue 中的 `bounty` 关键词。实际上 GitHub 赏金生态已有成熟中间层：

| 平台 | API | 说明 |
|---|---|---|
| **Algora** | `gh search issues --label="💎 Bounty"` | 最大的 GitHub 赏金平台，支持 USDC 支付 |
| **Opire** | `GET https://api.opire.dev/rewards` | 公开 API，无需认证，返回所有活跃赏金 |
| **Collaborators.build** | GraphQL 端点 | USDC 自动结算，merge 即付 |
| **Expensify** | `gh issue list` + `Help Wanted` 标签 | 明码标价 `[$250]` |
| **Gitcoin** | Grants + Bounties API | Web3 生态，金额较大 |

**建议**：新增 `bounty_aggregator.py`，聚合以上平台为统一的赏金 feed，标准化为：

```python
@dataclass
class Bounty:
    id: str
    source: str          # algora / opire / gitcoin / expensify
    title: str
    repo: str
    issue_url: str
    amount_usd: float    # 统一换算为美元
    currency: str        # USDC / USD / ETH
    labels: list[str]
    difficulty: str      # easy / medium / hard (AI 评估)
    competitors: int     # 已有多少 PR/评论
    created_at: datetime
```

### 3.2 反饱和策略（新增，核心竞争力）

当前 GitHub 赏金市场的问题是：**优质赏金发布后几分钟内涌入几十个 AI 生成的 PR**。成为第 11 个提 PR 的人，期望收益为零。

**差异化策略：**

| 策略 | 逻辑 | 优先级 |
|---|---|---|
| **废弃赏金收割** | 被 claim 但 14+ 天无 PR 的 bounty → 你上 | P0 |
| **拒绝修复翻新** | 看已有 PR 被拒原因（`CHANGES_REQUESTED`），针对性修复 | P0 |
| **零竞争窗口** | 监控赏金发布时间分布，找冷门时段发布的赏金 | P1 |
| **深水区专精** | 只做特定领域（Python 爬虫、数据处理、自动化脚本），靠质量而非速度 | P1 |
| **小额快反** | $10-50 的微赏金，大号看不上，AI 成本可覆盖 | P2 |

**建议**：在 `demand_radar.py` 的评分系统中增加 `saturation_score` 维度：

```python
def calc_saturation_score(issue_url: str) -> int:
    """0-10 分，越高越值得做"""
    prs = count_open_prs(issue_url)
    comments = count_attempt_comments(issue_url)
    staleness = days_since_last_activity(issue_url)
    
    if staleness > 14 and prs == 0:
        return 10  # 废弃赏金，最高优先级
    if prs == 0:
        return 9   # 零竞争
    if prs <= 2 and staleness < 3:
        return 6   # 可以一试
    if prs > 5:
        return 1   # 红海
    return 3
```

### 3.3 需求可解决性评估（新增，核心瓶颈）

当前评分只看"是不是付费需求"，不看"我能不能解决"。新增 `solvability_score`：

```python
def calc_solvability(issue_body: str, repo_context: dict) -> int:
    """评估 AI 能否解决此 issue"""
    # 硬过滤：明确不可自动化的
    HARD_BLOCKERS = [
        "need design review", "requires approval from",
        "security audit", "need access to internal system",
        "discuss first", "RFC", "proposal needed"
    ]
    if any(b in issue_body.lower() for b in HARD_BLOCKERS):
        return 0  # 无法自动化
    
    # 可自动化信号
    AUTO_FRIENDLY = [
        "add test", "fix bug", "update dependency",
        "refactor", "add feature flag", "improve error message",
        "migrate to", "update docs", "add type hint",
        "add validation", "fix typo", "optimize"
    ]
    score = sum(1 for s in AUTO_FRIENDLY if s in issue_body.lower())
    return min(10, score * 2)
```

## 四、需求解决层设计（Solve）—— 核心新建模块

### 4.1 架构

```
solver/
  __init__.py
  engine.py          # 解决引擎主控
  planner.py         # 需求 → 实施计划
  coder.py           # 实施计划 → 代码
  tester.py          # 代码 → 测试验证
  packager.py        # 代码 + 测试 → 可交付物
  templates/         # 常见需求类型的解决模板
    web_scraper.md
    data_pipeline.md
    api_wrapper.md
    bug_fix.md
    test_coverage.md
```

### 4.2 解决流程

```
Issue → [Planner] → Plan → [Coder] → Code → [Tester] → Pass → [Packager] → Ready
                          ↓ 人工审核点（可选）       ↓ Fail → 重试/放弃
```

**Planner**：用 LLM 将 issue 描述转化为结构化实施计划：
```json
{
  "files_to_modify": ["src/utils/fetcher.py", "tests/test_fetcher.py"],
  "approach": "add retry logic with exponential backoff",
  "estimated_loc": 30,
  "risk_level": "low",
  "test_strategy": "add 3 test cases for retry scenarios"
}
```

**Coder**：基于计划和 repo 上下文生成代码。关键——不要只靠 LLM 盲写，要：
1. 先读取目标文件
2. 理解现有代码风格
3. 生成 patch
4. 用 `ast` / `ruff` 做语法检查

**Tester**：运行项目的测试套件，确保不引入回归。关键——沙箱执行：
```python
# 在 Docker 容器中执行，防止恶意代码
def run_in_sandbox(repo_path: str, test_command: str) -> TestResult:
    ...
```

### 4.3 成本控制

这是商业模式的核心约束。一个 $25 的赏金，不能花 $5 的 API 费用。

| 模型选择 | 用途 | 成本 |
|---|---|---|
| 本地 ollama (qwen2.5-coder:7b) | 代码补全、简单 bug fix | $0 |
| Claude Haiku / DeepSeek | 计划生成、复杂逻辑 | ~$0.01/次 |
| Claude Sonnet | 高风险复杂需求，人工审核后使用 | ~$0.15/次 |

**成本上限**：单次解决总 API 成本 < 赏金额的 15%。

## 五、交付层设计（Deliver）

### 5.1 GitHub PR 自动交付

```python
class PRDeliverer:
    def deliver(self, solution: Solution) -> DeliveryResult:
        # 1. Fork repo (如果还没 fork)
        # 2. Clone to temp dir
        # 3. Create branch: forager/fix-{issue_number}
        # 4. Apply changes
        # 5. Push
        # 6. Create PR with structured description:
        #    - What was changed
        #    - How to test
        #    - References issue #
        # 7. Comment on issue: "Submitted PR #X addressing this"
```

### 5.2 交付后监控

提 PR 后不是结束——需要跟踪：
- PR 是否被 review
- 是否有 `CHANGES_REQUESTED` → 触发自动修复
- PR 是否被 merge → 触发赏金确认
- 赏金是否到账 → 记录收益

```python
@dataclass
class PRTracker:
    pr_url: str
    status: str          # open / changes_requested / merged / closed
    last_check: datetime
    auto_fix_attempts: int
    bounty_claimed: bool
```

### 7.2 `gpu_earner.py` — 重新定位

**当前问题**：此文件与核心商业闭环无关。Vast.ai 出租是独立收入线，但和"解决需求"没有关联。

**建议**：
- 保留 Vast.ai 状态查看功能（GPU 闲置时出租是合理的被动收入）
- 增加 `solver.py` 调用入口，替代现有的 `cmd_scan()`（扫描后自动送入解决引擎）
- 新增 `cmd_forage()` —— 完整的一次觅食循环：`scan → filter → solve → deliver`

```python
def cmd_forage():
    """完整觅食循环"""
    # 1. 扫描赏金
    bounties = BountyAggregator().scan_all()
    
    # 2. 过滤：我能解决吗？
    candidates = [b for b in bounties 
                  if b.solvability_score >= 7 
                  and b.saturation_score >= 6]
    
    # 3. 排序：性价比最高
    candidates.sort(key=lambda b: b.amount_usd * b.solvability_score / (b.competitors + 1), 
                    reverse=True)
    
    # 4. 解决（取前 1 个，保守策略）
    if candidates:
        solution = Solver().solve(candidates[0])
        
    # 5. 交付
        if solution.tests_pass:
            PRDeliverer().deliver(solution)
```

## 八、风险与对策

| 风险 | 概率 | 影响 | 对策 |
|---|---|---|---|
| **PR 质量低被社区骂** | 高 | 账号被封、声誉受损 | 设置最低置信度阈值、人工抽查、白名单 repo |
| **AI 生成代码引入 bug** | 高 | PR 被拒、浪费 review 资源 | 强制跑目标 repo 的测试套件、沙箱验证 |
| **赏金平台 anti-bot** | 中 | 账号被封 | 速率限制、模拟人类行为（随机延迟、工作时间窗口） |
| **法律/合规** | 低 | 侵权纠纷 | 只接 MIT/Apache 等宽松许可证的 repo |
| **API 成本失控** | 中 | 入不敷出 | 硬限制：单次解决 API 成本 < 赏金的 15% |
| **赏金不到账** | 中 | 白干 | 只做平台托管的赏金（Algora/Opire），不做私下承诺的 |

## 九、执行路线图

### Phase 1：验证最小闭环（1-2 周）

**目标**：手动选 1 个 GitHub bounty，用 AI 辅助解决并提 PR，验证整个流程跑通。

- [ ] 手动在 Algora 找一个 $20-50 的简单 Python bug fix
- [ ] 用 Claude 辅助写 fix + 测试
- [ ] 手动提 PR
- [ ] 记录整个流程的耗时、API 成本和结果
- [ ] **成功标志**：PR 被 merge，赏金到账

### Phase 2：自动化 Find 层（2-3 周）

**目标**：`bounty_aggregator.py` 上线，自动发现高价值赏金。

- [ ] 实现 Algora + Opire + Collaborators 的数据聚合
- [ ] 实现反饱和评分
- [ ] 实现可解决性评估
- [ ] 数据库 schema 升级
- [ ] **成功标志**：每天自动发现 10+ 个可自动解决的赏金

### Phase 3：自动化 Solve 层（3-4 周）

**目标**：`solver/engine.py` 上线，能自动解决简单 issue。

- [ ] 实现 Planner → Coder → Tester 流水线
- [ ] 支持 3 类需求模板：bug fix、add test、simple feature
- [ ] 沙箱测试环境
- [ ] 成本控制硬限制
- [ ] **成功标志**：AI 独立解决 50% 以上的 $10-50 简单赏金

### Phase 4：自动化 Deliver 层（1-2 周）

**目标**：`deliverer.py` 上线，自动提 PR + 跟踪状态。

- [ ] 自动 fork → branch → push → PR
- [ ] PR 状态跟踪
- [ ] CHANGES_REQUESTED 自动修复
- [ ] 收益记录
- [ ] **成功标志**：端到端自动化跑通一次

### Phase 5：规模化 & 风控（持续）

- [ ] 多语言支持（TypeScript、Rust、Go）
- [ ] 每日收益仪表盘
- [ ] 异常检测（连续失败 → 自动暂停）
- [ ] 人工审核队列（低置信度需求进队列）
- [ ] A/B 测试不同模型组合的成本/成功率

## 十、一句话总结

**Forager 的核心不是"爬取需求"，是"把 GitHub 赏金变成可自动化的流水线"。** 当前 `demand_radar.py` 是 good start，但它只是眼睛。需要加上手（solver）和嘴（deliverer），才能真的把食物带回来。

第一个 milestone 不是 100% 自动化——是先手动跑通一个 bounty，赚到第一块钱，验证这个闭环在物理世界是通的。之后每一层自动化都是在这个已验证的闭环上加杠杆。
