## 二、三个致命断点（含修复）

### 断点 1：`--forage` 只展示不执行

**位置**：[forager.py:396-435](forager.py)

**现象**：`cmd_forage()` 扫描赏金 → 评分 → 打印 top5 → 结束。没有调用 solver，没有调用 deliverer。

**修复**：

```python
def cmd_forage(scorer: Scorer, auto_deliver: bool = False):
    db = get_db()
    tracker = EarningTracker()
    
    # 1-2. Scan + Score (不变)
    agg = BountyAggregator()
    bounties = agg.scan_top(n=20)
    candidates = [(b, scorer.score(...)) for b in bounties if ...]
    
    # 3. 对 top1 实际执行 solve → deliver
    if candidates:
        top_bounty, top_score = candidates[0]
        console.print(f"[green]执行: {top_bounty.title[:60]}[/]")
        
        engine = SolverEngine()
        solution = engine.solve(
            title=top_bounty.title,
            body=top_bounty.body or top_bounty.title,
            repo=top_bounty.repo,
            issue_url=top_bounty.issue_url,
            bounty_amount_usd=top_bounty.amount_usd,
        )
        
        if solution.code and solution.tests_pass:
            deliverer = PRDeliverer()
            result = deliverer.deliver(
                repo=top_bounty.repo,
                issue_number=_extract_issue_number(top_bounty.issue_url),
                changes=solution.file_changes,      # ← 改为结构化的文件变更
                title=top_bounty.title,
                description=f"AI-generated fix for {top_bounty.issue_url}",
            )
            if result.success:
                tracker.record_found({...})
                tracker.mark_delivered(top_bounty.id, result.pr_url)
                console.print(f"[green]PR 已提交: {result.pr_url}[/]")
```

**关键变化**：`--forage` 从"展示推荐"变为"对最佳候选执行一次完整的 Find→Solve→Deliver"。

### 断点 2：solver 没有真正生成代码

**位置**：[solver/engine.py:55-62](solver/engine.py)

**现象**：
1. `engine.py` 调 `POST /codegen` 到 Miser localhost:7860 — 此端点不存在
2. `coder.py:generate()` 方法只返回一个 prompt 字符串，从未被 engine 调用
3. 生成的代码是裸字符串，没有指定要修改哪个文件的哪一行

**修复方案**（两种路径，推荐 A）：

**路径 A（推荐 — 最快跑通）**：直接调 Claude API 生成 unified diff

```python
# solver/engine.py 重写 solve() 方法

def solve(self, title, body, repo="", issue_url="", bounty_amount_usd=0) -> Solution:
    solution = Solution()
    solution.plan = self.planner.plan(title, body)
    
    # 成本检查
    est_cost = solution.plan.estimated_loc * 0.0003
    if bounty_amount_usd > 0 and est_cost > bounty_amount_usd * self.max_cost_ratio:
        solution.error = f"成本超限"
        return solution
    
    # 用 Claude API / DeepSeek 生成代码
    prompt = f"""You are fixing a GitHub issue.

Issue title: {title}
Issue body: {body[:1000]}

Output a JSON object with file changes:
{{
  "files": [
    {{
      "path": "relative/path/to/file.py",
      "action": "modify",   // modify | create | delete
      "content": "full new file content here"
    }}
  ],
  "test_command": "pytest tests/ -q",
  "explanation": "what was changed and why"
}}

Output ONLY the JSON, no markdown, no explanation outside the JSON."""

    import anthropic
    client = anthropic.Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])
    resp = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=4000,
        messages=[{"role": "user", "content": prompt}],
    )
    
    # 解析 JSON 输出
    result = json.loads(resp.content[0].text)
    solution.code = json.dumps(result)
    solution.file_changes = result.get("files", [])
    solution.tests_pass = True  # 实际测试在 deliver 阶段做
    
    return solution
```

**路径 B（长期）**：用 Claude Code CLI 直接操作文件系统

```bash
# 让 Claude Code 自己读 repo、改代码、跑测试
claude --print "Fix issue #42: add retry logic to fetcher.py. 
  Read the file, make the change, run pytest, then git diff."
```

路径 A 成本可控（一次调用 ~$0.05-0.15），路径 B 效果好但需要 Claude Code CLI 环境。

### 断点 3：deliverer 不知道改哪个文件

**位置**：[deliverer.py:72-75](deliverer.py)

**现象**：
1. `code` 参数是裸字符串，`deliver()` 把它写到 `fix-{issue}.patch` 新文件
2. 实际需要的是：修改 `src/utils/fetcher.py` 第 42 行，不是新建一个 `.patch` 文件
3. solver 输出和 deliverer 输入之间没有约定的数据格式

**修复**：

```python
# deliverer.py — 重写 deliver() 的 apply changes 步骤

def deliver(self, repo: str, issue_number: int, 
            changes: list[dict],   # ← 结构化变更：[{"path": "x.py", "action": "modify", "content": "..."}]
            title: str, description: str = "") -> DeliveryResult:
    
    with tempfile.TemporaryDirectory() as tmpdir:
        # 1. Clone
        subprocess.run(["gh", "repo", "clone", repo, tmpdir], ...)
        
        # 2. Create branch
        subprocess.run(["git", "-C", tmpdir, "checkout", "-b", branch_name], ...)
        
        # 3. Apply changes — 按文件路径覆写
        for change in changes:
            filepath = Path(tmpdir) / change["path"]
            if change["action"] == "modify" or change["action"] == "create":
                filepath.parent.mkdir(parents=True, exist_ok=True)
                filepath.write_text(change["content"], encoding="utf-8")
            elif change["action"] == "delete":
                filepath.unlink(missing_ok=True)
        
        # 4. Run tests (if available)
        test_cmd = changes[0].get("test_command", "")
        if test_cmd:
            test_result = subprocess.run(test_cmd.split(), cwd=tmpdir, ...)
            if test_result.returncode != 0:
                return DeliveryResult(success=False, error="Tests failed, PR aborted")
        
        # 5. Commit + Push + PR (不变)
        ...
```

**关键变化**：solver 输出 `list[FileChange]`，deliverer 逐文件覆写 → 跑测试 → 提 PR。两端数据格式对齐。

## 三、端到端闭环的最小可执行版本

不改架构，只修三个函数，串起来：

```
用户执行: python forager.py --forage
    │
    ▼
[forager.py] cmd_forage()
    │  1. BountyAggregator.scan_top(20)   → 发现赏金
    │  2. Scorer.score() × 20             → 评分过滤
    │  3. 取 top1 → SolverEngine.solve()  → AI 生成代码         ← 新增
    │  4. PRDeliverer.deliver()           → fork→改文件→PR       ← 新增
    │  5. EarningTracker.mark_delivered()  → 记录                 ← 新增
    ▼
  终端输出: "PR 已提交: github.com/xxx/pull/42"
```

**先不要求 100% 自动化**。先做到：扫描到一个赏金 → 用 Claude API 生成修复代码 → 自动提 PR。哪怕需要人工 confirm 一下再提，这个闭环在物理世界也是通的。

## 四、新增模块建议（对比 v1 有调整）

v1 提出的新模块都实现了，但需要修复内容。以下是对已有模块的修改建议：

### 4.1 `forager.py` — 改 `cmd_forage()`

| 项 | 现状 | 改为 |
|---|---|---|
| 执行范围 | 展示 top5 推荐 | 对 top1 执行 solve→deliver |
| 人工审核 | 无 | `--forage --dry-run` 只看不做，不加参数才执行 |
| 错误处理 | 无 | solver 失败 / test 失败 / PR 失败 分别打日志 |

### 4.2 `solver/engine.py` — 改 `solve()`

| 项 | 现状 | 改为 |
|---|---|---|
| 代码生成 | POST 到不存在的 MISER/codegen | Claude API / DeepSeek 调真实 LLM |
| 输出格式 | 裸字符串 `solution.code` | 结构化 `solution.file_changes: list[dict]` |
| Planner | 纯关键词匹配 | 加一个轻量 LLM 调用来理解 issue 意图 |
| 模型选择 | 写死 Miser | 读 `config.yaml` 的 models 配置，三级 fallback |

### 4.3 `deliverer.py` — 改 `deliver()`

| 项 | 现状 | 改为 |
|---|---|---|
| 输入格式 | `code: str`（裸字符串） | `changes: list[dict]`（结构化文件变更） |
| 应用变更 | 写到新文件 `fix-{issue}.patch` | 按 path 覆写目标文件 |
| 测试验证 | 无 | 跑 changes 中指定的 test_command |
| 分支命名 | `forager/fix-{issue}` | 保持不变，加时间戳防冲突 |

### 4.4 `solver/__init__.py` — 加 `FileChange` 数据类

```python
@dataclass
class FileChange:
    path: str          # "src/utils/fetcher.py"
    action: str        # "modify" | "create" | "delete"
    content: str       # 完整新文件内容
    test_command: str  # "pytest tests/test_fetcher.py -q"
```

让 solver 输出 `Solution.file_changes: list[FileChange]`，deliverer 接收同样的类型。

### 4.5 `earning_tracker.py` — 加自动对账

```python
def auto_reconcile(self):
    """轮询已交付 PR 的状态，检测是否 merge 并记录到账"""
    for row in self.db["earnings"].rows_where("status = 'delivered'"):
        pr_status = PRTracker().check(row["pr_url"])
        if pr_status.state == "merged":
            self.mark_paid(row["id"], row["amount_usd"])
```

## 五、执行顺序

| 阶段 | 做什么 | 产出 | 时间 |
|---|---|---|---|
| **Phase 1** | 修 `solver/engine.py` — 接 Claude API 真正生成代码 | solver 能产出结构化代码 | 2-3 小时 |
| **Phase 2** | 修 `deliverer.py` — 接收结构化变更、跑测试、提 PR | deliverer 能提有效 PR | 2-3 小时 |
| **Phase 3** | 修 `forager.py --forage` — 串起三层的完整流水线 | 端到端跑通一次 | 1 小时 |
| **Phase 4** | 手动找一个 $20-50 的 Algora bounty 做端到端测试 | 第一个 PR 被 merge，钱到账 | 1-3 天 |
| **Phase 5** | 加 `--dry-run`、日志、错误恢复、成本统计 | 可放心让它在后台跑 | 1-2 天 |

**判断成功的唯一标准**：Phase 4 完成——一个真实的 GitHub bounty，Forager 发现了它，AI 写了修复代码，PR 被 maintainer merge，钱打到账上。
