## 三、新断点：Find→Solve 鸿沟

v2 修了三个代码断点，v3.1 实测暴露了第四个——它不在代码里，在**产品策略**里。

### 问题本质

```
能 FIND 到的      → 高额基础设施任务 ($2k-$9k)，需要深度仓库理解
能 SOLVE 的       → 简单 bug fix / typo / 小功能，在 GitHub 上通常没有 bounty 标签
```

**两条线几乎不重叠。** Forager 扫回来的是大金矿，但铲子只有指甲刀大。

### 这个鸿沟有多大

从 Algora 的 30 条数据来看：
- **solvability=3（全部 Algora）**：AI 无法独立解决，需要 clone + read + understand + modify + test，当前 solver 没有仓库上下文
- **pay=10（几乎全部 Algora）**：因为 `[\$Nk]` 格式触发最高层付费匹配
- **tech_match=2-4（大部分 Algora）**：这些任务的技术栈（Go、Rust、分布式系统）不在 Python 关键词列表里

**评分维度之间正在互相抵消。** pay=10 拉高了总分，但 solvability=3 的赏金不应该被 `--auto` 执行。

### 为什么 v3.1 没有发现

因为测试从来没跑到 solve 那一步。如果跑到了，会看到：
1. Claude API 被调用，消耗了 $0.10-0.50
2. 返回的 JSON 是"需要更多上下文，我无法确定文件路径"
3. file_changes 为空或为幻觉输出
4. deliver 阶段收到一个不存在的文件路径
5. 整个 pipeline 静默失败，钱烧了但没有产出

## 四、三个源头策略需要重定

### 4.1 Algora：从"主攻"降级为"远攻"

Algora 是真金库，但当前 solver 吃不下。不能直接对 top1 Algora 赏金跑 `--auto`。

**短期策略**：只做 Find + 人工筛选
- 扫描 → 评分 → 展示，但不自动 solve
- 人工挑出"看起来 AI 可能能解决"的（简单 bug fix、typo、小重构）
- 手动触发 solve → 人工 review → 手动提交

**中期策略**：等 solver 支持 `--with-context` 模式（见第五节）

### 4.2 V2EX：不是编程需求平台

V2EX 是**论坛**，不是赏金平台。那些帖子是"找人代做"的议价帖，需要：
- 私信沟通需求细节
- 报价协商
- 交付验收
- 微信/支付宝转账

这不是自动化代码生成能解决的场景。**V2EX 更适合做需求发现雷达**，给用户手动筛选，不给 `--auto`。

### 4.3 GitHub Issues：搜索策略重写

当前搜索：
```
label:"good first issue"+label:bounty+is:open+language:python
```

问题：同时有 "good first issue" 和 "bounty" 标签的 issue 极少。实测 0 结果。

**应该改为**：不去找"有 bounty 标签"的 issue，而是**直接扫 Algora 返回的 issue 中那些金额小、标签简单的**。

或者走另一个方向：跳过 bounty 平台，直接搜 **GitHub 上标签为 `bug` + `good first issue` + Python + 最近活跃** 的 issue，不管有没有赏金——先让系统跑通 Solve→Deliver 闭环，证明它能给真实项目提真实 PR。**先赚 star，再赚钱。**

## 五、Solver 需要"仓库上下文"能力

这是 v3 最重要的架构新增。

### 当前 solver 输入

```
title: "Enforce membership on saved view sharing"
body: "When a user shares a saved view, we need to verify..."
```

Claude 看到的是 200 字的问题描述，不知道这个仓库长什么样，不知道文件在哪，不知道代码风格。它只能猜。

### 需要的 solver 输入

```
title: "..."
body: "..."
repo_context:
  - 仓库目录结构 (tree dump)
  - issue 提及的文件内容 (read from repo)
  - 相关测试文件内容
  - 最近 10 个 merged PR 的 diff（了解代码风格和变更模式）
```

### 具体方案：`repo_context` 预取

在 `SolverEngine.solve()` 中加一步（在调用 LLM 之前）：

```python
def _fetch_repo_context(self, repo: str, issue_body: str) -> dict:
    """克隆仓库，提取 issue 相关的文件上下文"""
    import tempfile, subprocess
    from pathlib import Path
    
    context = {"files": {}, "structure": "", "recent_diffs": ""}
    
    with tempfile.TemporaryDirectory() as tmpdir:
        # 1. 浅克隆 (depth=1, 快速)
        subprocess.run(
            ["git", "clone", "--depth", "1",
             f"https://github.com/{repo}.git", tmpdir],
            capture_output=True, timeout=60
        )
        
        # 2. 目录结构 (前 200 行)
        tree = subprocess.run(
            ["git", "-C", tmpdir, "ls-tree", "-r", "--name-only", "HEAD"],
            capture_output=True, text=True, timeout=10
        )
        context["structure"] = tree.stdout[:3000]
        
        # 3. 从 issue body 中提取提及的文件路径，读取它们
        import re
        paths = re.findall(r'[\w/.\-]+\.(?:py|js|ts|rs|go|java)', issue_body)
        for p in paths[:5]:
            fpath = Path(tmpdir) / p
            if fpath.exists():
                context["files"][p] = fpath.read_text()[:2000]
        
        # 4. 最近 merged PR 的标题和变更量（了解项目风格）
        # (可选，需要 gh CLI)
    
    return context
```

然后将 `repo_context` 注入 Coder 的 prompt：

```python
prompt = f"""You are fixing a GitHub issue in the repository {repo}.

REPOSITORY STRUCTURE:
{repo_context['structure'][:2000]}

RELEVANT FILES:
{json.dumps(repo_context['files'], indent=2)[:3000]}

ISSUE:
Title: {title}
Body: {body[:1000]}

Based on the ACTUAL repository structure and files above, output a JSON with file changes.
..."""
```

**成本估算**：
- 浅克隆一个中等仓库：~5-10 秒
- LLM prompt 增量：~5000 tokens → ~$0.02 (Claude Sonnet)
- 总的 solve 成本从 $0.10 升到 $0.15-0.20

**收益**：solver 从"盲目猜测文件路径"变成"看到真实代码后修改"，代码可用率从 ~5% 提升到 ~40-60%。

## 六、评分引擎需要加"自动执行门禁"

当前 `cmd_forage()` 对 top1 无脑执行。需要加两重门禁：

### 6.1 最小 solvability 阈值

```python
# forager.py cmd_forage() 中
if auto_deliver:
    if top_s.solvability < 5:
        console.print(f"[yellow]solvability={top_s.solvability} < 5, 跳过自动执行[/]")
        console.print(f"[dim]原因: {top_s.reasons}[/]")
        return  # 不执行
```

这避免了对着 solvability=3 的 $8000 基础设施任务烧 API 费。

### 6.2 最大赏金上限

```python
MAX_AUTO_BOUNTY = 200  # 自动执行只碰 $200 以下的赏金
```

$8000 的赏金不值得冒全自动的风险——万一 PR 质量差，maintainer 会拉黑你。小额赏金竞争少、风险低，更适合验证闭环。

### 6.3 评分维度权重调整

当前权重：
```
pay=30%  tech_match=20%  solvability=30%  saturation=20%
```

对于 `--auto` 模式，应该用**不同的权重**：
```
pay=15%  tech_match=25%  solvability=40%  saturation=20%
```

自动执行时，AI 能不能解决比赏金大小更重要。

## 七、正确的端到端测试路径（重写 Phase 4）

v2 的 Phase 4 是："手动找一个 $20-50 的 Algora bounty"。目前来看这句话需要改。

### 不应该做的事

- ❌ 对 Algora 的 $8000 赏金直接跑 `--auto`（代码质量不够，可能被拉黑）
- ❌ 对 V2EX 帖子提 PR（对方没有仓库，没有 issue tracker）
- ❌ 对着 solvability=3 的任务烧 API 费

### 正确的第一步（本周）

**不碰 Algora。** 去 GitHub 手动找一个 **Python 项目 + `good first issue` + 最近 7 天活跃 + issue 描述清晰（有具体文件路径）** 的 issue。

不管有没有 bounty 标签，目标是验证 Solve→Deliver 这条链路能不能产出**被 maintainer 接受的有效 PR**。

具体操作：
```
1. 手动找到目标 issue URL
2. python forager.py --forage --dry-run   (用 stdin 输入 issue URL)
3. 审查生成的代码
4. 如果合理 → python forager.py --forage --auto
5. 等待 maintainer review
```

**判断标准**：PR 被 merge（免费的也接受）。这证明了 Solve→Deliver 代码路径是可用的。

### 第二步（下周）

回到 Algora，但加上门禁和 repo_context：
```
1. python forager.py --forage              (扫描，不加 --auto)
2. 人工从 top10 中挑一个 solvability>=5 且金额<=$200 的
3. python forager.py --forage --auto --issue=<chosen>
4. 审查 PR 质量，人工决定是否提交
```

### 第三步（两周后）

加 repo_context 后，对 solvability>=5 的赏金跑全自动。

## 十、一句话总结

**Forager 的 Find 引擎找到了正确的金矿（Algora），但 Solver 的铲子太小，Scorer 还一直把人往最大最硬的矿石那边引。** v3 的核心修正不是改代码，是改策略：(1) 先找软柿子捏，证明闭环通；(2) 给 solver 装一把更大的铲子（repo_context）；(3) 让评分系统学会避开它现在挖不动的石头。
